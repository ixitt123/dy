# 安全更新与发布说明

## 核心原则

- 启动软件只启动本地程序，不自动拉取 GitHub，不启动自动上传。
- 后台改动提醒只提示，不执行 `git add`、`commit`、`pull` 或 `push`。
- `main` 是正式稳定分支，禁止直接推送，只能通过 GitHub Pull Request 合并。
- `settings.json`、API Key、`.data/`、下载和生成素材始终只保存在本机。

## 平时只需要记住三个入口

### 启动软件

双击：

```text
启动.bat
```

它不会自动更新或上传代码。

### 发布已经完成的修复

确认 Codex 已经完成修改后，双击：

```text
同步项目.bat
```

它会自动完成：

1. 安装本地 pre-push 安全钩子；
2. 检查当前分支；
3. 暂存并记录候选快照；
4. 运行完整离线闸门；
5. 确认测试期间文件没有继续变化；
6. 提交完整改动；
7. 核对远程分支；
8. 再经过 pre-push 闸门后上传修复分支。

如果当前是 `main`，工具会自动拒绝直接上传。

### 更新经过审核的正式版本

双击：

```text
安全更新.bat
```

只有同时满足以下条件才会更新：

- 当前位于 `main`；
- 本机没有未提交改动；
- GitHub 可以正常连接；
- 更新可以快进完成，不需要覆盖本地内容。

否则工具会停止并提示，不会覆盖文件。

## 查看状态

双击：

```text
查看同步状态.bat
```

它会显示当前分支、文件状态、最近提交、安全策略和同步日志。

## 本地安全钩子

`启动.bat` 和 `同步项目.bat` 会自动执行：

```powershell
git config core.hooksPath githooks
```

通常不需要手动配置。第二台电脑第一次启动软件时也会自动配置。

## 可选的开机改动提醒

`安装开机自动同步.bat` 现在只安装“本地改动提醒”。它不会自动拉取、提交或上传。

不需要提醒时，双击：

```text
卸载开机自动同步.bat
```

## GitHub 正式发布

日常开发使用 `fix/*` 或 `dev` 分支。修复分支上传后：

1. GitHub 自动运行 `check-gate`；
2. 检查通过后才允许合并 Pull Request；
3. `main` 禁止直接推送和强制覆盖；
4. 候选版本实际运行 24 至 48 小时后，才提升为稳定标签。

## 第二台电脑

第二台电脑不要自动同步开发中的代码。需要正式更新时：

1. 确认没有本地修改；
2. 切换到 `main`；
3. 双击 `安全更新.bat`；
4. 更新完成后再启动软件。

如果工具显示黄色或红色提示，不要尝试强行覆盖，把窗口截图发给 Codex。

## 失败处理

- 测试失败：不会提交或上传，联系 Codex 修复。
- 文件在测试期间变化：自动停止，等修改结束后重新同步。
- GitHub 连接失败：本地提交仍保留，不会丢失。
- 出现冲突：自动停止，不执行强制覆盖。
- `settings.json` 被 Git 跟踪：安全闸门立即失败，禁止发布。

禁止使用：

```text
git reset --hard
git push --force
git add -f settings.json
```
