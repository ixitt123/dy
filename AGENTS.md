# Codex 协作规则

## 工作方式

- 这台电脑是主要编辑端，尽量在本机 Codex 中完成项目修改。
- GitHub 仓库 `https://github.com/ixitt123/dy` 是唯一远程源。
- 其他电脑可以同步和应急编辑，但不要让两台电脑长时间同时改同一个文件。
- 启动软件、打开 Codex 和 Codex Stop hook 均不得自动拉取、提交或推送代码。
- Codex Stop hook 只允许运行测试并报告结果。
- 如果同步失败、被跳过或发生冲突，先处理 Git 状态，再继续编辑。

## 安全同步

- `.codex/config.toml`：Codex 项目级 hooks 配置。
- `.codex/hooks/stop-sync.mjs`：有改动时只运行 `check:gate`，不提交、不上传。
- `安装开机自动同步.bat`：只注册本地改动提醒，不执行 Git 写操作。
- `同步项目.bat`：人工触发安全测试、快照核对、提交和修复分支上传。
- `安全更新.bat`：人工更新无本地改动的正式 `main`。
- `查看同步状态.bat`：查看 Git 状态和最近同步日志。
- `卸载开机自动同步.bat`：关闭并移除开机提醒。

后台规则：

- 不自动执行 `git add`、`git commit`、`git pull` 或 `git push`。
- 不允许后台静默改变磁盘代码。
- 检测到改动时只能提醒用户运行安全同步。

## Git 同步规则

每次完成代码修改后：

1. 确认当前不是 `main`，使用 `fix/*` 或 `dev` 分支。
2. 运行 `npm run check:gate`。
3. 确认测试期间文件快照没有变化。
4. 只提交本次完整功能切片。
5. 推送修复分支。
6. 通过 GitHub PR 和 `check-gate` 合并到 `main`。

## 开始工作前

任何开发任务开始前：

1. 检查 `git status --short --branch`。
2. 如果位于 `main`，先建立独立修复分支。
3. 如需更新正式代码，必须在工作树干净时人工执行安全更新。
4. 如果存在冲突，停止自动操作并先解决冲突。

## 开发原则

- 不允许直接删除核心代码。
- 修改前先分析影响范围。
- 优先保持向后兼容。
- 每次功能开发必须提交到 GitHub。

## 安全改动协议

所有代码修复、功能开发、重构、依赖升级和安全整改，默认使用仓库 Skill：`$douyin-safe-change`。

1. 必须从包含 `.git`、本文件和根 `package.json` 的仓库根目录开始工作。
2. 修改前记录当前分支、远程、已修改文件、未跟踪文件和子模块状态；这些既有改动属于用户，不得覆盖、暂存、还原或混入本次切片。
3. 先复现问题并定义完成条件，再用 `rg` 追踪调用方、路由、事件、SQLite 字段、设置项、导出和 UI 入口。
4. Bug 修复优先先写能稳定复现问题的回归测试，确认修复前失败、修复后通过；不得削弱断言来消除失败。
5. 每次只处理一个最小完整切片，禁止顺手重构、无关格式化、无证据抽象和未批准的依赖升级。
6. 必须保持下载、文案提取与改写、任务队列、SQLite、TTS、字幕、小黑、CS1 和 MoneyPrinterTurbo 现有契约；未经用户确认不得删除兼容路径或旧数据。
7. 修改后先跑针对性测试，再运行 `npm.cmd run check:gate`；涉及 UI/API 串联、生产线、持久化、导出、TTS 交接、字幕、小黑、CS1 或 MoneyPrinterTurbo 时，还必须运行 `npm.cmd run test:e2e`。
8. 测试后运行 `git diff --check`，并将 Git 状态与开始前快照对比，调查任何意外生成或修改的文件。
9. 未经用户明确授权，不执行 pull、commit、push、merge、stash、restore 或切换分支。
10. 不能在当前任务实际运行并通过的测试，不得报告为“已通过”；必须明确写出未运行、失败或阻塞原因。

## Code Review Rules

### 生产线兼容性

- 阻止会破坏下载、改写、任务队列、SQLite、TTS、字幕、小黑、CS1 或 MoneyPrinterTurbo 既有行为的改动。
- 确认后的 TTS 音频仍是 CS1 与 MoneyPrinterTurbo 的标准下游载荷；任何替换都必须有明确迁移和端到端证据。
- UI 文案、服务端规则、持久化字段、导出和测试必须同步，不能只修其中一个入口。

### 安全与数据边界

- 重点检查路径穿越、命令执行、XSS、SSRF、任意文件读取、超大请求体和仅在前端实施的安全校验。
- 阻止 API Key、Secret、Cookie、Token、`settings.json`、SQLite、本地素材、下载和导出文件进入提交。

### 回归证据

- Bug 修复必须有能够保护用户可见行为的回归证据；若无法自动化，必须说明原因和人工验证路径。
- 拒绝通过删除测试、跳过测试、放宽关键断言或无解释改写预期结果来“修复”失败。

## 备份与发布

- 修改完成并通过测试后，应及时推送修复分支，避免只保存在本地。
- 自动备份若以后恢复，只能写入 `backup/电脑名`，不得写入 `dev` 或 `main`。
- `main` 禁止直接推送、强制推送和自动提交。

## 输出要求

每次任务完成后输出：

- 修改文件列表。
- Git 提交信息。
- Push 结果。
- 是否同步成功。

## 多电脑协作规则

1. GitHub 的 `main` 是唯一正式代码源。
2. 第二台电脑只在人工执行 `安全更新.bat` 时更新正式版。
3. 禁止两台电脑自动互相传播开发中的半成品。
4. 同一功能同一时间只在一台电脑和一个修复分支修改。
5. Push 失败或出现冲突时禁止强制覆盖。
6. 每次任务完成后报告分支、Commit ID、Push、PR 和 CI 状态。

## 禁止上传的内容

不要把这些内容提交到 GitHub：

- `settings.json`
- `.data/`
- `downloads/`
- `node_modules/`
- SQLite 数据库和临时文件
- API Key、Secret、Cookie、Token
- 本地生成的大体积素材和导出文件

这些内容应该保留在各自电脑本地。

## 项目架构优先规则

- `AGENTS.md`
- `skills/`
- `prompts/`
- `config/`
- MCP

架构优先。

禁止直接堆代码。

先设计 Skill。

再设计 Prompt。

最后开发功能。

## TTS 国内供应商优先规则

1. 本项目 TTS 默认优先使用国内供应商。
2. 第一优先级为阿里云百炼 CosyVoice / Qwen-TTS。
3. 火山引擎豆包语音、腾讯云 TTS 依次作为国内扩展；海外 Provider 只作为扩展，不作为默认方案。
4. API Key、Secret 只允许保存在本地 `settings.json`。
5. 不允许在日志、导出文件、错误信息中暴露 API Key 或 Secret。
6. 语音克隆必须要求用户确认拥有声音授权。
7. 所有 TTS Provider 必须通过统一 Adapter 接口调用。
8. 不允许把某个平台 API 调用逻辑写死在前端。
9. 不允许破坏现有下载、文案提取、AI 改写、任务队列、SQLite 功能。
10. TTS 文案必须先经过 Prompt 约束的清洗、切句、情绪和停顿处理，再交给 Provider。

## 声音资产中心规则

1. 不为声音资产中心继续增加 Provider，统一复用现有 TTS Adapter。
2. 参考音频、克隆资料和试听音频默认只保存在本机 `voices/`，不得自动上传 GitHub。
3. 声音克隆必须再次确认用户拥有合法授权。
4. 声音资产、测试记录、评分和默认音色必须写入 SQLite，不允许只保存在前端。
5. 预设音色、克隆音色、收藏、最近使用和默认音色必须使用同一套资产数据。
6. 自动测试必须使用 `prompts/voice_test_script.md` 的五类脚本，不允许把测试文案写死在页面。
7. 默认音色可被 AI 改写后的“一键生成语音”流程复用。
8. 版本管理不得覆盖旧版本；新版本必须保留父版本关系。

## AI 导演系统规则

1. Director System 第一阶段只生成专业导演稿，不生成图片或视频。
2. 不允许直接把文案随机切成图片提示词；必须先分析传播结构、情绪和镜头目的。
3. 所有分镜必须服务于文案情绪、传播目标和目标平台。
4. 禁止廉价 PPT 风、AI 图库拼接风、企业宣传片模板风和土味招生广告风。
5. 每条视频必须保持视觉风格、色调、字幕和镜头语言统一。
6. 每个镜头必须说明 purpose，不允许无意义空镜和无意义转场。
7. Director Prompt 必须放在 `prompts/`，不得写死在业务代码里。
8. Director Skill、风格库、镜头语言和审美规则必须放在 `skills/`。
9. Director 项目、镜头、状态和评分必须写入 SQLite，并通过独立队列处理。
10. 不允许破坏现有下载、文案提取、AI 分析、AI 改写、TTS、声音资产中心和 SQLite 功能。

## APS 与 VFO 规则

1. Asset Planning System 只负责素材规划，不生成图片或视频，不调用图片 Provider。
2. VFO 只负责决策、调度、规划和质量控制，不直接生成图片或视频。
3. 工作流固定为：Storyboard → APS → VFO → Image Provider → Video Renderer → QA → Export。
4. 每个镜头必须说明素材类型、选择原因、生成或获取方式、渲染策略和质量检查项。
5. 素材分类、平台策略、渲染策略和 QA 规则必须来自 `skills/`、`prompts/` 和 `config/`，不得写死在前端。
6. APS 必须输出 Asset Package、Asset Review 和 Render Readiness。
7. VFO 第一阶段只输出 `render_plan.json`，不接图片 Provider、视频 Renderer 或自动导出成片。
8. 抖音、视频号、小红书和 B 站必须使用各自的平台信息密度、字幕、镜头时长和画幅策略。
9. APS 与 VFO 项目、镜头、评分和状态必须写入 SQLite，并通过独立队列处理。
10. 不允许破坏现有下载、提取文案、AI 改写、Director、TTS、声音资产中心及其 SQLite 数据。
